## Learned User Preferences
- User prefers Dutch-language collaboration and phrasing in this workspace.
- User wants explicit provenance tracking for skills, including GitHub source grouping and credits.
- User values direct command-style triggers for skills (for example creating dedicated slash-style skills for immediate use).
- User expects continual-learning runs to follow strict incremental-index rules and exact output contracts when no updates are found.
- User prefers command examples to be shown as terminal-style blocks with prompts and aligned presentation.
- User prefers design choices to be explored with side-by-side visual examples before implementation.
- User prefers English for committed repository documentation; personal operational runbooks (for example continual-learning and npm release workflows) live under `.cursor/private/`, which is gitignored.
- User wants bundled `core/` skills and public examples to assume multiple Chain Hub–linked editors: generalize guidance where the capability is shared, state explicit scope when something is Cursor-only or otherwise product-specific, and include the full supported editor set in lists or examples where relevant (including Antigravity).
- User wants the published chainhub.one documentation (including `/docs`) aimed at end users of the CLI and hub (how to use Chain Hub and clear up behavior), not internal developer-only checklists in the shipped site build.
- User prefers the marketing homepage hero and compact footer without repeating the public domain as primary navigation (tagline and GitHub link suffice; omit prominent “domain · Docs” style rows there).
- User intends optional add-on offerings (for example a premium skill pack) to ship and update on a separate track from protected `core/` contents bundled in the npm package, and is exploring hosted Cloud Sync for teams or multi-device hubs as part of that Pro-oriented track.
- User wants the local Chain Hub dashboard served by `chain hub` (`apps/hub`) to feel visually consistent with the landing-site branding and to adapt cleanly across screen sizes (for example a persistent sidebar on desktop and mobile-friendly navigation such as a drawer), with the logo beside the name, without a redundant host (for example `localhost`) subtitle under the title, and with hub styling favoring Tailwind utility classes over heavy bespoke CSS layers where practical; visual polish of the Skills list (scanability and hierarchy) is a priority for that UI.

## Learned Workspace Facts
- `CHAIN_HOME` resolves in this order: `--chain-home`, `CHAIN_HOME` env, `chain config set chain_home` (persisted in `~/.config/chain-hub/config.json` or `$XDG_CONFIG_HOME/chain-hub/config.json`), then default `~/chain-hub`; commands like `init`, `status`, and `validate` show the active path and source. A dev shell alias that binds `chain` to `bun run …/cli/chain.ts` shadows the npm global binary and can cause missing-`chain.ts` / `Module not found …/chain.ts` if the path is stale after moving the repo; prefer `CHAIN_HUB_REPO` + `alias chain="bun run $CHAIN_HUB_REPO/cli/chain.ts"` (see `cli/README.md`) or remove the alias when testing the published package.
- Marketing site (`apps/web/`) editor symlink examples are generated from `cli/adapters/setup-display-paths.ts` so they stay aligned with `cli/adapters/*.ts`; CI runs `bun run check:adapter-docs` after the web build to catch drift.
- When a Chain Hub home includes `skills-registry.yaml`, it can track ownership via `authorship.self`, registry buckets (`core` for bundled skills mirrored under `skills/`, `chain_hub` for `chain add` from the index, `personal`, etc.), and GitHub bundle provenance under `github_sources` with credits.
- The published npm package is `chain-hub`; manifest and versioning live in `cli/package.json` — run `npm publish`, `npm pack`, and `npm version` from `cli/` (the monorepo root has no `package.json`). The executable is `chain`. Marketing site https://www.chainhub.one/ and `/docs` are built with Astro from `apps/web/` (`bun run build` → `apps/web/dist/`). The `chain hub` dashboard is a static Astro app in `apps/hub` (Tailwind CSS v4 via `@tailwindcss/vite`; `bun run build` in `apps/hub` → `apps/hub/dist/`), copied to `dist/hub` as part of `cli` `bun run build`; from a source checkout `chain hub` serves `apps/hub/dist` when present (so run `bun run build` in `apps/hub` after UI changes, or use `bun run dev` there to run `astro dev` on port 4321 with `/api` proxied to `chain hub` on 2342).
- Shipped protected content lives in repository `core/` (skills, rules, agents, workflows, `registry.yaml`, and `core/templates/` such as shadcn scaffolding); `chain init` copies `core/` into `CHAIN_HOME` and syncs listed protected agents, workflows, and rules into `CHAIN_HOME/agents/`, `CHAIN_HOME/workflows/`, and `CHAIN_HOME/rules/` for IDE adapters.
- **Single user hub:** `CHAIN_HOME` (default `~/chain-hub`) is the canonical location for all data managed by Chain Hub: after `chain init`, `core/` holds the protected bundled copy; user and registry-installed content lives at the hub root in `skills/` (created empty on init), `agents/`, `workflows/`, `rules/`, and `skills-registry.yaml`. IDE adapters symlink *from* `CHAIN_HOME`; Universal links may also mirror into `~/.agents/`, but that path is not a second primary library root. User-defined agents and workflows sit beside protected slugs under `agents/` and `workflows/`; `chain list` separates protected core from user content where applicable. Protected slugs listed under `rules:` in `core/registry.yaml` are mirrored from `core/rules/` into `CHAIN_HOME/rules/` during `ensureCoreAssets` (for example `chain init`, `chain setup`, and the hub dashboard **Fix** / **Fix all** actions, which call the same setup path as `chain setup` via `/api/setup`) so bundled rule symlink targets exist; `chain status` may still report broken IDE symlinks for other missing paths or mis-pointed links. That setup path relinks IDE symlinks and refreshes core assets; it does **not** run **`chain fix`** (frontmatter / skill-workflow text repair) and does not create arbitrary files beyond what `ensureCoreAssets` and adapter link targets cover.
- `chain add` accepts registry slugs and `github:owner/repo`, not arbitrary local directories; copy or symlink a local skill into `CHAIN_HOME/skills/<slug>/` and register the slug in `skills-registry.yaml` (for example under `personal:`) or `chain validate` fails. Registry installs from `registry/index.yaml` copy the full indexed skill directory from the GitHub tarball (not only `SKILL.md`). Skills expect a flat layout (`skills/<slug>/` only); nested category folders under `skills/` are not supported by validation or standard IDE symlinks.
- CI and releases: `.github/workflows/ci.yml` runs on every push and PR: in `cli/`, `bun install --frozen-lockfile`, then `bun install --frozen-lockfile` in `apps/hub` (needed for the Astro hub build inside `pack:check`), then `bun test`, `bun run pack:check`, `bun run smoke:package`, and `bun run smoke:hub`; in `apps/web/`, `bun run build` and `bun run check:adapter-docs`. Changelog via git-cliff (`cliff.toml` at repo root; `cli/` scripts `changelog` / `changelog:preview`) includes conventional `feat` / `fix` / `revert` for `cli/**` and `core/**` only. Until **1.0.0**, updating `cli/CHANGELOG.md` is optional; from **1.0.0** onward, run `bun run changelog` in `cli/` before publishing. For npm pack/publish (notably npm 11+), `package.json` `bin` paths must not use a `./` prefix (for example `dist/chain.js`, not `./dist/chain.js`); otherwise npm may drop the `bin` entry as invalid.
- `chain reflect` runs `distillLearnings()` on the active `CHAIN_HOME` and, when the learnings inbox has queued events, writes distilled drafts under `learnings/drafts/`; use `--dry-run` to print what would be distilled without writing. On a Chain Hub **source** checkout (`cli/chain.ts` and `core/registry.yaml` present), `chain setup` may install `.git/hooks/pre-commit` to run `chain validate` with `CHAIN_HOME` set to that repo; the hook uses the `chain` binary on `PATH` and skips if `chain` is missing.
- Validator allows reserved skill names for registered upstream GitHub sources while still rejecting custom reserved names.
- Syncing Chain Hub without `.git/` across Syncthing/Dropbox-like tools can propagate large `skills/` deletions from reduced inventories.
